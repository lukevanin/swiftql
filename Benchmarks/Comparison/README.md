# Cross-library SQLite full-fetch baseline

This directory preserves a reproducible before/after reference for SwiftQL's
current large result-set path. It measures one operation: open a read-only
connection once, then fetch and decode all 16,143 rows and all 14 columns from
Northwind's `Orders` table. It is integration evidence for future execution
work, not a general ranking of database libraries.

## Provenance and licensing

The workload idea was surveyed in Lighter's separately hosted
`PerformanceTestSuite` at revision
`892dfa0bb8fc023a8a25de873f34c3fb766c927a`. That repository has no license at
the pinned revision. This harness therefore records it only as provenance: it
does not clone or execute the suite and does not copy its source, harness
structure, comments, or generated artifacts.

The benchmark driver and adapters here were independently implemented against
the compared libraries' public APIs. Lighter 1.4.12's MIT-licensed Enlighter
plugin generates accessors during each isolated graph build from this
repository's fixture; generated Swift is neither copied from upstream nor
checked in.

The database snapshot derives from the MIT-licensed
`jpwhite3/northwind-SQLite3` repository at revision
`4f56e7f5906dfd23b25244c5bfe8fb5da6402efd`. The source population script is
nondeterministic, so the committed compressed artifact—not the Git revision or
a random seed—is authoritative. The runner verifies the archive and database
hashes, byte count, integrity, 4,096-byte page layout, exact `Orders` columns,
and row count before building. See [Fixtures/README.md](Fixtures/README.md) and
[Fixtures/Northwind-LICENSE.txt](Fixtures/Northwind-LICENSE.txt).

This 16,143-row performance snapshot is deliberately separate from issue
#254's pinned 830-order correctness corpus.

## Compared paths

Two release executables are required because SwiftQL currently resolves GRDB 6
while SQLiteData resolves GRDB 7. Each executable contains the same six
controls plus its graph-specific implementation:

| Identifier | Measured path |
| --- | --- |
| `generated_raw_sqlite` | Raw SQLite accessors freshly emitted by Enlighter. |
| `lighter` | Accessors freshly generated by Lighter's Enlighter plugin. |
| `grdb_manual` | GRDB rows mapped field by field. |
| `grdb_codable` | GRDB's `FetchableRecord`/`Decodable` path. |
| `sqlite_swift_manual` | SQLite.swift raw-statement values mapped field by field. |
| `sqlite_swift_typed` | SQLite.swift's typed `Table`/`Expression` query path. |
| `swiftql` | SwiftQL table/result decoding through its GRDB 6 adapter. |
| `sqlite_data` | Point-Free's SQLiteData table fetch through GRDB 7. |

The runner always selects the same 14 columns and checks exactly 16,143 decoded
rows after every fetch. A full 14-field checksum is computed outside timing for
the first and final fetch in each process, and those checksums must match.
Checking only those two complete results avoids adding a large string-hashing
thermal workload between every timed sample while still observing every field.

### Schema-mapping differences

SwiftQL, SQLiteData, both GRDB paths, and both SQLite.swift paths use `String?`
for the three `DATETIME` columns and `Double?` for `Freight`. The typed
SQLite.swift query explicitly casts `Freight` to `REAL` because SQLite may
store integral values in the fixture's `NUMERIC` column as `INTEGER`; its
manual path accepts either storage class.

Enlighter's native mapping is materially different: the generated model uses
`Date?` for the three date columns and `Decimal?` for `Freight`. Both the
high-level Lighter and generated-raw paths retain those conversions. This is
an intentional native-mapping comparison, not a type-normalized decoding
contest, and the Date/Decimal cost must be considered when interpreting their
numbers.

## Measurement method

Both dependency graphs are fully resolved and release-built before timing,
followed by a 180-second idle cooldown. Every implementation then runs in its
own fresh process. That process opens its connection and prepares its adapter,
performs 10 unrecorded warmup fetches, then records 100 individual full-fetch
durations with `DispatchTime.uptimeNanoseconds`.

Both executable targets use Swift 5 language mode under the recorded current
toolchain. Lighter 1.4.12's generated database singleton is not concurrency-safe
under Swift 6 language checking; using Swift 5 mode avoids modifying or
checking in generated third-party output and is applied identically to both
graphs.

There are three independent processes per implementation. The process schedule
rotates the six controls and runs matching controls in adjacent graph pairs;
the first graph alternates by process. The JSON retains UTC start/finish times
and a contiguous schedule index so the validator can prove the actual order.

The headline median is the median of the three process medians. The headline
p95 is the median of the three nearest-rank process p95 values. Process spread
is `(maximum process median - minimum process median) / headline median`.
Rows per second is `16,143 / median seconds`.

Shared-control drift is the symmetric percentage difference between the two
graph medians. A report is rejected above 5%. The guard is deliberately larger
than the 1.5–3.5% control noise observed while correcting the process schedule;
differences below 5% are treated as noise, not as evidence of a winner.

On macOS, each isolated process is wrapped with `/usr/bin/time -l`. Peak RSS is
the largest of the three reported process peaks. It includes the executable,
its dependency graph, connection, warmups, and retained result memory; it is
not an incremental allocation attributed only to the API. Other platforms
must record an explicit unavailability reason.

## Reproduce the baseline

Start from a clean SwiftQL checkout. The workspace must be new or empty, and
the output file and sibling `Runs` directory must not already exist:

```sh
python3 Benchmarks/Comparison/run.py \
  --workspace /private/tmp/swiftql-comparison \
  --swiftql-checkout "$PWD" \
  --output Benchmarks/Comparison/comparison-results.json \
  --cooldown-seconds 180
```

The runner verifies the exact fixture, copies only repository-owned templates
into the empty workspace, builds both pinned release graphs, preserves every
sample/resource log, validates paired-control drift, and writes the v3 JSON.
Use `--prepare-only` to inspect the independently prepared graphs without
building or timing them. Use `--allow-dirty` only for exploratory work; that
state is recorded and is not suitable for a committed baseline.

Validate and render a report using only the Python standard library:

```sh
python3 Benchmarks/Comparison/summarize.py \
  Benchmarks/Comparison/comparison-results.json
```

Compare a compatible candidate while allowing only the SwiftQL source revision
to differ:

```sh
python3 Benchmarks/Comparison/summarize.py \
  --baseline Benchmarks/Comparison/baseline.json \
  --candidate Benchmarks/Comparison/candidate.json
```

Validation rereads the fixture/license artifacts and every raw TSV/resource
log, verifies their hashes and records, recomputes all derived statistics,
requires the complete timestamped schedule and process set, checks dependency
pins and environment compatibility, and enforces the paired-control guard.

differences on those paths as conclusive.

An initial run with the original 60-second cooldown was correctly rejected:
its paired GRDB Codable drift was 5.047%, just above the 5% guard. No report or
raw files from that attempt are committed. The accepted result came from a new
workspace and the longer 180-second cooldown; the threshold was not changed.

The run used a Mac16,8 with Apple M4 Pro, 14 cores, 24 GiB memory, arm64 macOS
26.5.1 (25F80), Xcode 26.5 (17F42), Swift 6.3.2, and system SQLite 3.51.0.

sample TSVs and 42 `/usr/bin/time -l` resource logs under [`Runs/`](Runs/),
preserving all 4,200 timed samples and their verified SHA-256 values.

## Interpretation and limits

This case measures a warmed, repeated full-table fetch and typed materialization
on one machine. It includes SQLite stepping and each API's row/model creation;
it excludes package compilation, connection creation, the first ten warmups,
result correctness/checksum work, writes, predicates, joins, pagination,
concurrency, observation, and application-level processing.

The two graphs necessarily contain different GRDB majors and transitive
dependencies. Paired controls quantify that graph-level noise but cannot make
the graphs physically identical. Generated raw access is a useful workload
control, not a theoretical hardware or SQLite lower bound.

Do not extrapolate this one schema and access pattern to a general library
ranking. Broader runtime workload design belongs to issue #259. Consumer
compile time, generated-source size, binary size, and a dedicated RSS matrix
belong to issue #257. No absolute timing or memory value in this directory is
a CI gate.

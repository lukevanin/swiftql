# SwiftQL v1.3 release-readiness audit

Audited on 19 July 2026 against the live
[v1.3 milestone](https://github.com/lukevanin/swiftql/milestone/8) and
`origin/main` at `4f2af35eefc97cd8db03e2740f83454ff85e4980`.

## Verdict

**Ready for milestone closure.** SwiftQL's v1.3 conformance, correctness,
compatibility, documentation, agent guidance, and release-process evidence
agree with the existing SQLite-surface boundary in `ROADMAP.md`. No product,
correctness, compatibility, packaging, security, documentation, performance,
or release-process blocker remains for closing issue
[#226](https://github.com/lukevanin/swiftql/issues/226) and milestone 8.

This is not a claim that `v1.3.0` has been published. The changelog correctly
remains `Unreleased`, both the remote `v1.3.0` tag and GitHub Release were
absent, and
[`v1.2.0`](https://github.com/lukevanin/swiftql/releases/tag/v1.2.0) remained
the latest published package at audit time. Post-milestone release preparation,
tagging, publication, and independent verification are tracked outside
milestone 8 by
[#298](https://github.com/lukevanin/swiftql/issues/298).

### Shipped boundary and migration record

- `Package.swift` retains `swift-tools-version: 5.9`, iOS 16, macOS 13, and
  the `SwiftQLCore`, `SwiftQL`, and diagnostic `swiftql-benchmark` products.
  Its dependency lower bounds remain SwiftSyntax 509.0.0, GRDB 6.29.3, and the
  Swift-DocC plugin 1.0.0.
- `SwiftQLCore` remains the GRDB-free contract layer. `SwiftQL` remains the
  application-facing typed DSL, macros, codecs, and GRDB-backed SQLite adapter.
  SwiftQL does not claim another public dialect, driver, Linux runtime, Swift 6
  language mode, asynchronous cursor, nested transaction API, or migration
  engine in v1.3.
- The public query, binding, descriptor, codec, row-lifetime, and
  prepared-handle ownership contracts introduced in v1.2 remain unchanged.
  `CHANGELOG.md` therefore states that no migration is required for v1.3.
- `README.md`, the DocC catalog, `COMPATIBILITY.md`, `CHANGELOG.md`,
  `RELEASING.md`, and the root `SKILL.md` agree on the public boundary.
  `README.md`, the DocC catalog, `RELEASING.md`, and the root `SKILL.md`
  explicitly identify `1.2.0` as the latest published package.
- The #132 targets are package-private research targets, not public products.
  v1.3 ships no public build validator, SwiftPM validation plugin, query macro,
  schema system, persisted SQLite statement, or zero-runtime-parse mechanism.

### Compatibility and release controls

- The compatibility workflow has seven release-blocking compiler cells: Swift
  5.9 and Swift 6.0 committed and clean resolution, plus Swift 6.1, 6.2, and
  6.3 clean resolution. With release-tooling and source-coverage jobs, a full
  compatibility run contains nine required jobs.
- The compiler cells validate exact toolchain/SDK families, dependency
  resolution, first-party warnings, SQLite provenance, full tests, strict
  concurrency, and applicable downstream, conformance, and stress gates.
- The public GitHub API reported `v1.2.0` as the latest published,
  non-prerelease, immutable release. The `v1.3.0` tag and release were absent.
- Public ruleset record 19095830 reported exactly one repository-owned,
  active `Protect v-prefixed release tags` rule for `refs/tags/v*`, with no
  exclusions and only update- and deletion-blocking rules.
- The public API does not expose the repository immutable-release setting,
  ruleset bypass actors, or the current user's bypass status. #298 therefore
  requires fresh authenticated verification of those properties immediately
  before tagging. That is a publication preflight, not evidence that this
  implementation milestone is incomplete.
- `scripts/ci/check-release-readiness.sh` deliberately preserves the
  historical v1.1-only server-side gate and is not v1.3 milestone evidence.

# SwiftQL v1.4.6 release-readiness audit

Milestone [v1.4.6 (#14)](https://github.com/lukevanin/swiftql/milestone/14),
"Hardening and release closeout." This is the audit deliverable for
[#229](https://github.com/lukevanin/swiftql/issues/229). It is an evidence-backed
record, not an implementation task, and it does not weaken, skip, or relabel any
validation to reach a verdict.

## Verdict

**Not ready for milestone closure.** The hardening and documentation work in the
milestone landed on `version/1.4.6` and validates green, but four items block a
clean closure:

1. **#266's headline latency goal is not met.** The merged change reduces
   incremental-decode allocation and preserves the bounded RSS, but a controlled
   before/after shows it is latency-neutral; the regression is decode-compute
   bound. The remaining latency is owned by follow-up
   [#353](https://github.com/lukevanin/swiftql/issues/353). #266 must be
   explicitly accepted-as-partial (with #353 carrying the goal) or held.
2. **The milestone description over-scopes its issues.** It promises "live-query
   resume semantics," but the owning issue
   [#291](https://github.com/lukevanin/swiftql/issues/291) is scheduled in
   **v1.5.0 (#22)**, not v1.4.6. The description and the issue set must be
   reconciled.
3. **The supported-surface index [#115] is intentionally open** (21/33 children,
   63%) and, by its own closing rule, needs an explicit disposition before the
   milestone can "close with no open issues."
4. **No release is prepared.** The latest published tag is `v1.4.3`; `CHANGELOG`
   carries an untagged `[1.4.4]` section and no `v1.4.5`/`v1.4.6` entry. Release
   preparation and publication are a separate, later step (see `RELEASING.md`)
   and have not run.

None of these is a correctness or validation failure. The blockers are goal
completion (#266), planning-metadata reconciliation, and the not-yet-run release
step. Recommended dispositions are in
[Remaining work and disposition](#remaining-work-and-disposition).

### #166 — construction/rendering overhead

Delivered against its acceptance criteria. A checked-in deterministic
allocation profiler (`swiftql-construction-profile`, `malloc_logger`-based)
attributed ~77% of the `swiftql_construction_and_rendering` phase's allocations
to rendering; three output-preserving builder changes reduced render allocations
~15% and the phase's median ~13–17% across four cases, with byte-identical SQL,
743 tests passing, and warnings/strict-concurrency clean. Evidence:
[`Benchmarks/Profiling/Issue166`](../../Benchmarks/Profiling/Issue166/README.md).
No rendered SQL, entity metadata, public 1.x API, or Swift 5.9 support changed;
no absolute CI latency gate was added.

### #266 — incremental-decode latency (partial)

The merged change reuses one per-row normalization buffer instead of allocating
a fresh `[XLSQLiteValue]` (and a redundant `Array(...)`) per row, preserving the
#248 bounded-memory guarantee (no `[[XLSQLiteValue]]` matrix; copy-on-write keeps
the eager compatibility shims correct). A SwiftQL-graph before/after on the exact
16,143-row fixture measured **76.08 ms → 76.22 ms (+0.2%, inside the 0.3% base
spread)** with peak RSS **~37 MiB preserved** — a genuine null result. The #248
regression is therefore **not** allocation-bound; the dominant per-row cost is
SwiftQL's typed-decode compute (the harness's `grdb_manual` control does the
same fetch in ~6.9 ms). Evidence:
[`Benchmarks/Comparison/Issue266`](../../Benchmarks/Comparison/Issue266/README.md).
The remaining latency work is scoped atomically in
[#353](https://github.com/lukevanin/swiftql/issues/353). #266's latency goal is
**not** met by this milestone.

## Remaining work and disposition

Every item below is tracked; there are no untracked TODOs.

1. **#266 latency goal — deferred, tracked.** The decode-compute latency is owned
   by the atomic follow-up [#353](https://github.com/lukevanin/swiftql/issues/353)
   (labelled P3, with the #250-harness contract). Disposition: accept #266 as
   partial for v1.4.6 with #353 carrying the goal, or hold the milestone for a
   quiet-machine profiling pass. Owner decision required.
2. **Milestone description vs. issues — reconcile.** The v1.4.6 description names
   "live-query resume semantics," but the owning issue #291 sits in v1.5.0 (#22).
   Disposition: edit the v1.4.6 milestone description to drop the live-query
   clause (it is v1.5.0 scope), or move #291 if it was misfiled. Recommended:
   correct the description — #291's own contract is explicitly the "v1.5 async
   observation foundation."
3. **#115 tracking index — disposition.** Per its own rule it "closes only when
   every planned v1.4 child is complete or explicitly moved to a named later
   milestone" (21/33 done). Disposition: keep #115 as a standing index outside
   the release-blocking set, or retarget it to a later milestone, so v1.4.6 can
   satisfy `RELEASING.md`'s "no open issues" gate.
4. **Roadmap vs. milestone — reconcile #43.** `ROADMAP.md` lists "direct scalar
   CTE rows (#43)" under v1.4, but #43 is not in the v1.4.6 milestone. Disposition:
   confirm whether #43 is shipped, deferred, or belongs to a later milestone, and
   align the roadmap row with the milestone.
5. **Release preparation — not started (expected).** Latest published tag is
   `v1.4.3`; `CHANGELOG` has an untagged `[1.4.4]` and no `v1.4.5`/`v1.4.6`
   section. When the owner approves closure, prepare the release strictly per
   `RELEASING.md` (which owns changelog dating, version, tag, and the verified
   publication workflow). #166's internal performance improvement is a candidate
   `Performance`/`Changed` changelog note at that time; it required no API change.

The verdict returns to **ready** once #266 is accepted-as-partial (or completed),
items 2–4 are reconciled, and the release step is authorized. If any validation
listed above fails on re-run, the verdict is **not ready** until the failure is
fixed; it is never relabelled to obtain a ready verdict.
